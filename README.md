# Design System Kit

Готовый к переносу кит дизайн-системы.

Проще всего открыть в Cursor пустую папку и попросить сначала **« Склонируй в этот проект https://github.com/codenol/new_design_kit_llm.git , ветку dasha »** (у каждого своя ветка, читай ниже).

Потом попросить **«Инициализируй проект»** и получить экран-витрину компонентов.

## Ветки команды (кто где работает)

**Зачем это нужно.** В GitHub у проекта есть несколько «дорожек» — **веток**. У каждого участника своя ветка: так правки не перепутываются, а общая линия — **`main`** (часто её называют «мастер») — собирается осознанно.

**Кому какая ветка**

| Кто | Ветка (имя в GitHub / в Cursor) |
|-----|----------------------------------|
| Даша | `dasha` |
| Саша | `sasha` |
| Ириша | `irisha` |
| Лена | `lena` |
| Серёжа | `serezha` |

**Серёжа (ветка `serezha`)**

- Рабочая ветка Серёжи — **`serezha`** (переключение в Cursor так же, как у дизайнеров: внизу слева выбрать ветку).
- **Серёжа может мёржить в `main`** — то есть вливать готовые изменения в общую «мастер»-ветку, когда пора.
- **Серёжа проверяет MR остальных** — merge request’ы (в GitHub они называются **Pull requests**) от дизайнеров и других участников смотрит и принимает/отклоняет он.

**Что сделать один раз после того, как проект уже открыли в Cursor**

1. Внизу слева в Cursor нажмите на название ветки (часто там написано `main`).
2. В списке выберите **свою** ветку из таблицы выше — например, Даше нужна **`dasha`**.
3. Дождитесь переключения. Дальше все изменения в файлах (стили, макеты в коде, картинки в проекте) сохраняются **в вашей ветке**.

**Правила простые**

- Работайте **только в своей** ветке — не переключайтесь на чужую без договорённости.
- Ветку **`main`** дизайнерам лучше не трогать напрямую: это общая «чистая» линия; **в неё вливает Серёжа** (мёрж из своей ветки или после проверки MR).
- Если непонятно, что нажать, или Cursor спрашивает про «merge» / «conflict» — **остановитесь** и напишите Серёже: это нормально, вам помогут.

**Как отдать работу**

- Сохраняйте файлы как обычно (`Ctrl+S`). Когда порция правок готова — напишите в чат команды, что ветка готова к проверке; дальше **Серёжа** посмотрит MR (pull request) или заберёт изменения из GitHub и подскажет, как сделать «коммит».

Если вы дизайнер и **вообще не пользуетесь GitHub** — достаточно помнить: **одна девочка = одна ветка из таблицы** (не перепутать с Серёжей — у него `serezha`), переключить ветку в Cursor по шагам выше, и не лезть в чужие.

## Что внутри

- **design-system/** — токены, layout, helpers, overrides, uikit (см. design-system/README.md).
- **.cursor/rules/** — правила для Cursor: при запросе «Инициализируй проект» собрать витрину.
- **.cursor/skills/init-design-system/** — навык с пошаговой инструкцией сборки витрины.
- **src/** — минимальное приложение: главная страница — витрина (сайдбар, хлебные крошки, фильтры, таблица с бейджами и статусами, меню).

## Запуск (проверка в текущем проекте)

```bash
cd design-system-kit
npm install --legacy-peer-deps
npm run dev
```

Откроется витрина на http://localhost:5174 (или указанный порт).

## Перенос в новый проект

1. Скопировать всю папку `design-system-kit` в новую директорию.
2. Открыть её в Cursor.
3. Выполнить `npm install --legacy-peer-deps` (или `npm i` при совместимых версиях).
4. При необходимости написать **«Инициализируй проект»** — агент по правилам и навыку соберёт или обновит витрину.
5. Запустить `npm run dev` и при необходимости менять экраны и логику под свой проект.

## Проверка и доработка

- Проверять витрину можно здесь же: запустить dev-сервер из папки кита и открыть в браузере.
- Править Rules и Skills в `.cursor/rules/` и `.cursor/skills/`.
- Стили и компоненты — в `design-system/` и `src/`. После изменений перезапустить dev или обновить страницу.
