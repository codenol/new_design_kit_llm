# Сборка экрана по дизайн-системе

## Когда применять

Пользователь просит **«собери экран "…"»**, **«построй экран …»**, **«сделай страницу …»** в проекте на базе копии design-system-kit. Правило **build-screen** вызывает этот навык для пошаговой сборки.

## Цель

Провести короткий опрос как аналитик (с предложением вариантов ответов), затем собрать новый экран только из компонентов дизайн-системы, при нехватке компонентов предложить несколько вариантов реализации, добавить маршрут и пункт меню, запустить приложение и показать экран в браузере Cursor IDE.

## Шаги

### 1. Опрос (аналитик)

Задать по очереди (или компактно) вопросы и **предложить варианты ответов**:

- **КТО пользуется экраном?**  
  Варианты: администратор системы, оператор, наблюдатель (только просмотр), гость.
- **ЧТО пользователь делает на экране?**  
  Варианты: просмотр списка, создание/редактирование записей, фильтрация и поиск, массовые действия, выбор элемента для перехода.
- **ЗАЧЕМ нужен экран, какой результат?**  
  Варианты: мониторинг состояния, настройка параметров, отчёт, выбор сущности в мастере.
- **Ограничения?**  
  Варианты: только чтение, нужен экспорт/печать, мобильная версия, без бэкенда (моковые данные).

Дождаться ответов пользователя (хотя бы кратких) перед генерацией.

### 2. Определить блоки экрана и компоненты ДС

По ответам определить:

- Нужен ли список (таблица/дерево), форма, фильтры, детализация, действия (кнопки, меню).
- Сопоставить с каталогом: layout (AppLayout), uikit (StatusBadge, Tree, Message, BreadCrumbs, DataTableDynamic, DialogTabs, Grid, Drawer, StepWizard, Accordion и др.), PrimeReact (Button, InputText, Dropdown, DataTable, Menu и др.). См. `design-system/components/README.md` и папки в `design-system/uikit`.

### 3. Нехватка компонентов — несколько вариантов

Если под задачу нет готового компонента в ДС — сформулировать **два или более варианта** реализации (например: «Вариант А: DataTable с фильтрами и моковыми данными; Вариант Б: Tree слева и панель деталей справа»). Спросить пользователя или реализовать первый вариант по умолчанию.

### 4. Реализация экрана

- Создать страницу (например `src/pages/Nodes.tsx` или по названию экрана).
- Использовать **только** компоненты и стили из design-system: обёртка через `AppLayout` с `breadcrumbTrail` и `children`, классы из layout, uikit и PrimeReact (overrides).
- Данные — моковые (массив в состоянии), если бэкенд не предусмотрен.
- Добавить маршрут в `src/App.tsx` (или где объявлены `Routes`), например `<Route path="/nodes" element={<Nodes />} />`.
- Добавить пункт меню в сайдбар: в `src/layout/AppLayout.tsx` в массив `mainNav` (или в общий список навигации) — новый пункт с `to="/nodes"` и подходящей иконкой/подписью.

### 5. Запуск и показ в браузере

- В корне проекта выполнить `npm run dev` (при необходимости проверить `package.json` для имени скрипта и порта).
- Дождаться вывода с URL (например `http://localhost:5173`).
- Открыть в браузере Cursor IDE (MCP cursor-ide-browser) полный URL нового экрана (например `http://localhost:5173/nodes`).
- При необходимости сделать snapshot или screenshot для проверки отображения.

## Компоненты и классы

- Layout: `AppLayout`, `breadcrumbTrail`, классы `layout-wrapper`, `layout-sidebar`, `layout-main`, `layout-menu`, `router-link-exact-active`.
- uikit: StatusBadge (code: ok, warning, critical, info, …), BreadCrumbs, Tree, Message, DataTableDynamic, DialogTabs, Grid, Drawer, StepWizard, Accordion и др.
- PrimeReact: Button, InputText, Dropdown, DataTable, Column, Menu, BreadCrumb, Checkbox, InputSwitch, Dialog, Panel, Accordion, Tooltip.
- Бейджи: классы `status-badge`, `status-badge--ok`, `status-badge--warning` и т.д.

## Результат

Новый экран открывается по своему пути, отображается в общем layout с сайдбаром и хлебными крошками, собран только из компонентов ДС. Проект запущен, экран показан в браузере Cursor IDE.
