# Feature Analysis: Модель здоровья ПАК (Platform Application Kit)

## Actors
- **System Admin** - администратор системы, отвечает за мониторинг здоровья компонентов
- **Developer** - разработчик, использует Health Dashboard для отладки и диагностики
- **Operator** - оператор системы, следит за состоянием компонентов в реальном времени

## Entities
- **HealthStatus** - общий статус системы (healthy/degraded/unhealthy/unknown)
- **Component** - компонент системы с атрибутами:
  - `id` - уникальный идентификатор
  - `name` - название компонента
  - `type` - тип компонента (database, cache, queue, api, etc.)
  - `status` - текущий статус (healthy/degraded/unhealthy/unknown)
  - `lastCheck` - время последнего проверки
  - `metrics` - метрики (cpu, memory, disk, network)
  - `errors` - список последних ошибок
- **Metric** - метрика компонента:
  - `cpuUsage` - использование CPU (%)
  - `memoryUsage` - использование памяти (%)
  - `diskUsage` - использование диска (%)
  - `networkLatency` - задержка сети (ms)
  - `activeConnections` - активные соединения
- **Alert** - предупреждение/инцидент:
  - `id` - уникальный идентификатор
  - `type` - тип (critical/warning/info)
  - `message` - сообщение об ошибке
  - `component` - затронутый компонент
  - `timestamp` - время возникновения
  - `status` - статус (active/resolved)
  - `severity` - уровень критичности
- **HealthCheck** - результат проверки здоровья:
  - `componentId` - ID проверяемого компонента
  - `timestamp` - время проверки
  - `result` - результат проверки
  - `duration` - время выполнения проверки
  - `details` - дополнительные детали

## User Flows
1. **Обзор общего статуса**
   - Пользователь открывает Health Dashboard
   - Отображается общий статус системы (badge с цветом)
   - Показывается время последнего обновления
   - Отображается сетка компонентов с их статусами

2. **Просмотр деталей компонента**
   - Пользователь кликает на компонент в сетке
   - Открывается модальное окно с детальными метриками
   - Показываются графики/таблицы с историей метрик
   - Отображаются последние ошибки (если есть)

3. **Реакция на предупреждения**
   - Пользователь видит активные предупреждения в разделе Alerts
   - Критические предупреждения подсвечиваются красным
   - Пользователь может просмотреть детали инцидента
   - При разрешении инцидента статус обновляется

4. **Фильтрация и поиск**
   - Пользователь фильтрует компоненты по типу или статусу
   - Поиск по названию компонента
   - Сортировка по времени последнего обновления

5. **Экспорт отчёта**
   - Пользователь экспортирует текущее состояние системы
   - Формируется JSON/CSV отчёт со всеми метриками

## Edge Cases
- **Нет данных о компоненте** - компонент не отвечает, статус `unknown`
- **Частые проверки** - компонент проверяется слишком часто, возможна нагрузка
- **Прерванная проверка** - проверка была прервана, результат не записан
- **Метрики недоступны** - некоторые метрики не могут быть получены
- **Множественные ошибки** - компонент в состоянии `unhealthy` с несколькими ошибками
- **Переход статусов** - компонент переходит из одного статуса в другой (нужна анимация)
- **Оффлайн режим** - система работает без подключения к сети
- **Большие объёмы данных** - много метрик, нужна пагинация/виртуализация
